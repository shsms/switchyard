* Roadmap

** B. Browser-driven UI tests

The Rust integration tests in =tests/= cover the HTTP and gRPC
boundaries but stop at the JSON shape — none of them exercise the
SPA. The dashboard panel, the topology canvas's drag / connect
gestures, the REPL's autocomplete + electric-pair + rainbow
overlay, and the chart-update path through the WebSocket are all
unverified by anything other than manual smoke. The plan below
turns that into automated coverage.

Tool choice: *Playwright* (Node, headless Chromium / WebKit /
Firefox, auto-wait built in, native screenshot diff). It's the
default for projects without a JS framework — the alternative
(Cypress) is Chrome-mostly and heavier; Selenium is verbose. One
=playwright.config.ts= + a =tests-browser/= directory.

Determinism is the real pain. The topology graph layout is
seeded but not reproducible across machines (font metrics,
vis-network's animation timing); the physics tick + history
sampler push WS frames continuously; consumer-curve / cloud-curve
animations move power around at 200 ms. Tests need to either
freeze the world at known points or assert on stable structure
rather than pixel-exact screenshots.

** P. Physics realism upgrades

The electrical layer is deliberately behavioral (no power flow, no
losses) — see the architecture notes in CLAUDE.md. The two items
below are the cheapest upgrades that a downstream control app's own
accounting would actually notice; island-mode frequency coupling
(swing equation) was considered and deferred — not needed while
every scenario runs grid-tied.

*** TODO P1. Battery round-trip efficiency + idle draw             :fidelity:

Today charging and discharging both move SoC at 100 % efficiency and
an idle battery holds SoC forever. A control app that schedules
charge/discharge cycles and audits the energy balance sees numbers a
real site never produces. Add to =BatteryConfig=:

- =:efficiency-pct= (round-trip, default ~92): apply √η on each leg —
  charging stores =P·dt·√η=, discharging draws =P·dt/√η= — so a full
  cycle loses the configured fraction and the loss is visible on
  whichever leg the app measures.
- =:idle-draw-w= (default a few W): constant self-discharge +
  BMS/electronics draw integrated every tick regardless of setpoint,
  so SoC sags slowly over a long soak run.

Both default-off-able for configs that want the old ideal behaviour
(=:efficiency-pct 100 :idle-draw-w 0=). SoC integration stays in
=Battery::tick=; the meter aggregation path is untouched (losses are
internal to the battery, the inverter still reports commanded AC).

Bounds caveat (from the 2026-06-04 review): the gateway's AC∩DC bounds
intersection in =aggregate_child_bounds= is only numerically correct at
η=1. Once efficiency lands it must account for the conversion, or the
setpoint gate validates against mismatched AC-side and DC-side envelopes.

*** TODO P2. Battery voltage-vs-SoC curve                          :fidelity:

=dc_voltage_v= is a constant from config, so =dc_current_a=
(= apparent/V) never varies with state of charge — real packs sag
toward the bottom and rise toward the top of the SoC window, and
BMS dashboards key on exactly that. Replace the constant with a
simple piecewise-linear curve parameterised by (=:voltage-min=,
=:voltage-nominal=, =:voltage-max=) anchored at (soc_lower, 50 %,
soc_upper), keeping =:voltage-v= as the nominal for backward
compatibility. Telemetry-only change — power physics stays on W, so
no setpoint / bounds behaviour shifts.

** High

*** TODO [#A] Cross-microgrid eval corruption under concurrent per-mg eval :concurrency:

=with_microgrid= (=src/sim/microgrids.rs=) sets the shared
=current_microgrid= pointer *before* =eval= takes the interpreter lock,
and =eval= re-reads =current= for both =overrides_path()= and
=router.site()=. Two concurrent =/api/mg/{A}/eval= + =/api/mg/{B}/eval=
(e.g. two browser tabs) cross: A's mutations and appended overrides land
on B. =with_microgrid= also fails to restore the pointer if =f()= panics.

Fix: make scope-set + eval atomic under one lock, or thread an explicit
=mg_id= through =eval= / =overrides_path= / =router.site= instead of
ambient global scope; convert =with_microgrid= to an RAII guard so the
pointer is restored on unwind.

*** TODO [#A] Active and reactive setpoints share one timeout + reset :bounds:

=do_set_power= (=src/server.rs=) calls =add_timeout(id, …)= for both
active and reactive setpoints, and =TimeoutTracker= is keyed by id only.
=reset_setpoint= (the inverters) resets the active ramp *and* the
reactive controller together. So a reactive setpoint's expiry snaps
active power to idle, and setting either kind replaces the other's
pending deadline — but they're independent commands in the proto.

Fix: key the tracker by =(id, SetpointKind)= (the enum already exists in
=server.rs=) and make =reset_setpoint= take the kind so only the expired
surface resets.

** Medium

*** TODO [#B] Unbounded recursion on a topology cycle             :hardening:

Power aggregation (=children_of= in =src/sim/microgrid_site.rs=, via
=Meter::aggregate_active= and the inverters) has no visited-set or depth
cap, and =connect= / =disconnect= aren't gated by the =ComponentGraph=
validator. A user-introduced cycle (=(connect a b)= + =(connect b a)=)
makes =telemetry()= / =tick= recurse to a stack overflow that aborts the
site.

Fix: thread a visited-set / depth limit through the aggregation walk, or
reject cycle-creating edges in =connect=.

*** TODO [#B] CommandDelay doesn't re-clamp an armed setpoint        :bounds:

=BatteryInverter::tick= (and =SolarInverter::tick=) clamp the pending
target to the envelope only on the tick a command is freshly armed; a
later-arriving tighter augmentation / SoC-derate never pulls the armed
target back in, so the ramp drives toward an out-of-envelope value. The
EV charger already handles this.

Fix: in the =delay.poll == None= branch re-clamp =ramp.target()= to the
current =effective()= envelope and =set_target= if it changed, mirroring
=ev_charger=.

*** TODO [#B] persisted_overrides does file I/O under the interpreter lock :concurrency:lisp:

=persisted_overrides= / =persisted_count= (=src/lisp/mod.rs=) hold the
=ctx= write lock across =parse_file= (blocking I/O), reachable from sync
HTTP handlers. That's the lock the 100 ms refresh loop and every eval
contend on, so a slow / large overrides file stalls physics-input
updates for every microgrid.

Fix: parse the overrides file with a throwaway =TulispContext=, or read
the bytes first and hold the lock only for the in-memory parse.

*** TODO [#B] Lisp active-timers list grows unbounded across a long run :lisp:

=random-outage= (=sim/scenarios.lisp=) conses a fresh one-shot timer
handle onto =active-timers= each cycle and never prunes fired ones; only
=reset-state= (reload) clears it. A multi-day scenario accumulates
thousands of dead handles — the same slow-accumulation shape as the
edge-app incident.

Fix: prune fired one-shots when re-scheduling (filter on live status), or
reuse a single tracked handle slot per outage chain.

*** TODO [#B] Snapshot save/load ignore the per-request microgrid scope :concurrency:

=snapshots_save= / =snapshots_load= (=src/ui/handlers/snapshots.rs=)
resolve the target via the ambient =current_microgrid= (set by the last
per-mg eval) rather than a caller-specified id, so they can snapshot /
restore the wrong microgrid. Same root cause as the concurrent-eval
finding above.

Fix: add explicit microgrid-scoped snapshot routes and apply the same
atomic-scope fix.

** Fidelity

*** TODO [#B] Reactive config bounds sampled at live P, not rated — research first :reactive:fidelity:
=make_component_proto= (=src/proto_conv.rs:107=) fills the static
=AcPowerReactive= =MetricConfigBounds= from =c.reactive_bounds()=, whose
inverter impls sample at *live* P (=battery_inverter= =measured_w=,
=solar= =ramp.actual()=) — but the comment claims "full rated active P".
So an idle PF-limited inverter advertises =(0, 0)= reactive capability in
=ListElectricalComponents=, and a controller that caches the listing
concludes it can't do Q.

Research on the internet before fixing (not urgent): real inverters
differ — some are fully reactive-capable up to the kVA circle at any
active power (including P≈0), others can only source/sink Q as a
percentage of the *current* active power. Decide which model(s)
switchyard should expose and how the static config bounds (rated) vs the
live telemetry bounds should each read. Likely answer: config bounds use
rated P (=p_max=), telemetry keeps live-P. Check IEEE 1547-2018
reactive-at-zero-P capability and typical nameplate Q ratings.

*** TODO [#B] Reported PV bounds ignore sunlight curtailment :fidelity:bounds:
=SolarInverter::effective_active_bounds()= returns rated ∩ augmentation,
so telemetry advertises e.g. =[-30 kW, 0]= at 50% sun while the inverter
only delivers -15 kW. I1 made the *output* curtail to =min_avail_w=, but
the *bounds* still don't reflect irradiance, so a controller bidding max
generation sees headroom that isn't there. Fold =min_avail_w= into the
reported lower bound — but confirm against the real frequenz API (does it
report irradiance-limited inclusion bounds for PV?).

*** TODO [#B] kVA-circle not re-clamped when P drifts under a settled Q :reactive:
=ReactivePath::step= (=src/sim/reactive.rs=) only re-clamps Q to
=√(S²−P²)= when a command freshly promotes; if active power then drifts
under a sustained Volt/VAR setpoint, published apparent power exceeds the
rating until the next setpoint. Re-clamp the armed Q to the live envelope
each tick.

*** TODO [#B] dc_power_w is signed-apparent but its bounds are active-only :telemetry:reactive:
A battery's =DcPower= telemetry value is =√(P²+Q²)·sign(P)= while the
attached bounds are the active SoC-derated envelope (=battery.rs= /
=proto_conv.rs=). Value and bounds sit on different scales whenever Q≠0;
the scenario energy integrator also over-states charge/discharge Wh by up
to the PF headroom. Report active P as the value, or attach apparent
bounds.

*** TODO [#B] CHP and Grid stream no power telemetry of their own :telemetry:
=Chp= is a stub (parent meter carries =:power=) and =Grid::telemetry=
emits only id+category (the downstream meter carries flow). Faithful to
the "meter carries the literal" design, but a controller written to the
real API (subscribe by category = Grid / Chp) gets an empty metric set.
Document the convention at the API boundary, or synthesize an
=AcPowerActive= sample on these components.

*** TODO [#B] Telemetry state-code is decoupled from health       :telemetry:
=component_state= is derived from power sign (=power_state(p)= →
idle/charging/discharging); the ERROR =ElectricalComponentStateCode= only
comes from =TelemetryMode::ErrorEmpty= (=proto_conv.rs:292=). So an
=:health 'error= inverter (now zero-output after =27e079d=) still reports
state *idle*, indistinguishable from a healthy parked one, unless its
telemetry mode is also set. Decide whether health should drive an ERROR
state snapshot or stay orthogonal, and document it. (This is the open
part of I3.)

* Open design questions

** d1. Rethink ~file-exists-p~ resolving against the config's load dir :design:lisp:

The current =defuns::fs::register= (=src/lisp/defuns/fs.rs=) resolves
relative paths in =(file-exists-p PATH)= against the config file's
load directory (the same base =(load PATH)= uses). The motivation
was to keep =load-overrides='s gate

#+begin_src elisp
(when (file-exists-p "microgrids/config.7.overrides.lisp")
  (load "microgrids/config.7.overrides.lisp"))
#+end_src

answering the same question both halves of that =when= ask — the
two used to disagree when the process CWD differed from the config
dir, which broke the override-replay path inside
=remove_persisted_override= and the test that exercises it.

That's still a real coordination problem, but the fix is a little
heavy-handed:

- *Emacs divergence.* Real Elisp's =file-exists-p= resolves against
  =default-directory=, not the load path. Anyone porting code from
  Emacs (or eyeballing the docstring) would expect CWD-relative
  resolution; ours quietly rebases.
- *Surprises for non-override paths.* =(file-exists-p
  "telemetry.csv")= etc. now also rebases — fine if the user
  always means "next to config.lisp" but wrong if they meant CWD.
- *Coupling to =load_dir= at registration time.* The closure
  captures the path on =Config::new=, so a later =set_load_path=
  on the ctx (if we ever expose one to Lisp) wouldn't be visible
  to =file-exists-p=. =(load PATH)= in tulisp reads =ctx.load_path=
  on every call.

*Better directions to consider:*

1. Drop the special handling in =file-exists-p= and instead change
   the Lisp side: =load-overrides= in =sim/common.lisp= could build
   an absolute path before the gate, eg. by exposing the
   override-file path as a Rust-side defun (=(overrides-path)=
   already exists; make it return an absolute path).
2. Land a tulisp-level fix: add a =ctx.load_path= getter (currently
   =pub(crate)=) so a closure can resolve at call time, mirroring
   =(load PATH)=.
3. Push a =file-exists-p= change upstream into tulisp itself,
   matching =(load)='s resolution.

(2) or (3) is probably the right answer; (1) is the smallest patch
on the switchyard side.

*Where the workaround lives today:* =defuns::fs::register= closes
over =load_dir: PathBuf=, propagated from =Config::new= via
=defuns::register_runtime=. The matching test
(=ui::tests::persisted_remove_drops_form_immediately=) inlines a
local =load-overrides= defun and relies on this resolution to find
the override file.

** d2. Promote per-mg override file into the per-mg config file :design:lisp:ui:

Currently every successful eval lands in the per-microgrid override
file (=microgrids/config.<id>.overrides.lisp=). The next step in
that flow: a UI action that takes the override file's contents and
folds them into the matching =microgrids/config.<id>.lisp= (inside
the =:topology= lambda — see below), then empties the override
file. After the action the world is unchanged but the source-of-
truth has shifted from "runtime edit" to "canonical microgrid
config" and the entry is no longer editable from the UI overrides
dialog.

*Where it'd land:* a button in the chrome (probably called Persist
or Promote), a server-side endpoint that does the file munging, and
an entry in this UI dialog that confirms the move.

*Open questions to settle before coding:*

- Where in =microgrids/config.<id>.lisp= does the content go? The
  natural slot is inside the =(make-microgrid :topology (lambda
  () …))= lambda body, alongside whatever topology the file
  already declares. Insertion point: before the trailing
  =(load-overrides)= call, so the canonical forms run first and
  then any remaining overrides layer on top. Mechanical append at
  the end of the lambda is the floor — semantic grouping (collect
  rename-component calls near the renamed components, etc.) is
  much harder and probably not worth automating in v1.
- Does the action try to dedup against existing forms in the
  per-mg config file? eg. if the override is
  =(rename-component 1 "alpha")= and the config already calls
  =(rename-component 1 "main-grid")= inside the topology, do we
  replace the existing call, append after it, or just append at
  the end and let last-write-win semantics carry the day?
- Does the action run =tulisp-fmt= over the merged file, or
  preserve the existing whitespace / comments verbatim and only
  format the appended block?
- After the move, the override file is empty and the per-mg config
  file grew. Reloading should yield the same microgrid (same
  forms, just relocated). Worth a test that round-trips
  add → promote → reload → state-equal.

*Why deferred:* the immediate-write override file makes the
override file the pending log. Promote-to-config is a one-way
commit gate on top of that, but the design questions above need a
real config sample to test against — easier to settle once a heavy
microgrid config has accumulated some real overrides.

** d3. Parallel-paths share semantic — does this match real installs? :design:fidelity:

We currently model "child shared by N parents" as "each parent sees
1/N of the child's flow" (parallel-conductor semantic). It's
physically real for high-current buses split into N cables for
ampacity, each with its own CT and meter, summed by an upstream
aggregator. But that's an unusual SCADA-level topology.

The more common multi-parent setup in real microgrids is *redundant
observers* — two meters on the *same* wire (Class A + Class B,
primary + backup) both reporting the full flow, with the SCADA layer
deduping or selecting one. Under our current semantic, those two
meters would each read half, which isn't what the user expects.

If user feedback pushes us either way, options are:

- *Keep parallel-share as default*, document the redundant-observer
  use case as "express it via =:hidden t= on the backup meter so it
  doesn't appear in the connections graph".
- *Flip default to count-fully-per-parent*, and add an explicit
  =:share <fraction>= plist arg for the parallel-conductor case.
- *Add a topology-level kind annotation* (=:redundant-with <id>=,
  =:parallel-with <id>=) so the connection graph itself encodes
  intent. Most expressive, most invasive.

** d4. Per-phase reactive setpoints                         :design:reactive:

The proto's =SetElectricalComponentPower= takes a single scalar
=power= value, not per-phase. Real inverters can be commanded to
dispatch unbalanced reactive across phases for voltage support. Out
of scope for v1; would need a proto extension or a side-channel
defun. Tracked here so we don't forget the gap.

** d5. Per-source attribution under MxN                     :design:fidelity:

Today an inverter publishes its *commanded* AC output as =measured=;
the battery separately publishes the accepted (clamped) value. A
SCADA client that wants to see "inverter 1 produced 7.5 kW of the
13 kW total flowing into the shared battery" has to read both
streams and reconcile. A future upgrade: track per-source pushes on
the battery's accumulator (=Vec<(parent_id, p, q)>=), distribute
the clamp ratio across sources, and let each inverter read its own
=accepted_from(self.id)= to compute fair-share measured. Minor
trade-off in trait surface.

** d6. Persist only graph-affecting evals to the overrides file :design:lisp:

Today =Config::eval= (=src/lisp/mod.rs=) appends *every*
successful eval, verbatim, to the per-mg overrides file
(unconditionally on success), then calls =router.site().bump_version()=.
So a structural edit (=(make-battery …)=, =connect=, delete,
=rename=) and a transient runtime poke (=(set-meter-power 1 5000)=,
=(set-component-health 3 'error)=, =(+ 2 3)=, a defaults edit) all
land in the file identically. Because =(load-overrides)= replays the
file on reload, runtime pokes currently *resurrect* as if they were
config — so gating the append also fixes a correctness bug, not just
clutter. An =eval_silent= twin already evals without
persisting, so the "should this persist?" seam exists; it's just
all-or-nothing.

Structural mutations funnel through five =MicrogridSite= methods —
=register= (the =make-*= path), =connect=, =disconnect=,
=remove_component=, =rename=. Runtime/env defuns
(=set-component-health=, =set-frequency=, =set-*-power=, =every= /
=run-with-timer=) touch none of them, so "did the graph change?" is
observable. =bump_version()= fires on *every* eval — it's a
WS-refresh counter, not a structural signal.

*** The catch: "graph-affecting" is narrower than "worth persisting"

Two cases persist via the same eval path today but are *not* graph
edits:
- *Defaults edits* — the Defaults panel saves =(setq battery-defaults
  …)= through =/api/eval= (=ui-assets/dialogs.js=). They change
  *future* =make-*=, not the live graph. A strict graph-diff gate
  would drop them — a regression, since they're config worth keeping.
- *Renames* — change a component's display name only. Counts as the
  graph, or not? Open.

So the real decision is the *policy* (graph-structure only, vs.
"structural + config, minus transient/queries"), and only then the
mechanism.

*** Options

- *A — fingerprint-diff gate.* In =Config::eval=, hash the structure
  (=components()= → sorted ={id,category,subtype}= + =connections()= →
  sorted edges [+ names]) before/after; append only on change. Robust
  to any lisp (progn, loops, helper defuns) — decides on observed
  effect, not syntax. Cons: net-zero edits (add+remove in one form)
  don't persist; a mixed form (=make-*= *and* =set-power=) persists
  whole (runtime part replays); defaults dropped unless whitelisted.
- *B — structural-change-counter gate (recommended core).* Add
  =structural_version: AtomicU64= to =MicrogridSite=, bumped inside
  the five mutators; =Config::eval= snapshots it before/after and
  appends only on change. Keep =bump_version()= as the WS counter.
  Precise (knows a structural op ran), dirt-cheap (one atomic
  compare, no hashing), tiny surface. Cons: net-zero edits still
  persist (counter bumps twice); same mixed-form / defaults caveats
  as A. Can layer A on top to suppress net-zero.
- *C — verb allowlist (avoid alone).* Persist only if the eval's head
  is a graph verb. Brittle — =progn= / =dotimes= / =let*= / user
  defuns / =load= all defeat it. A hint, not the gate.
- *D — origin-based (good refinement).* Canvas actions
  (add/connect/delete/rename) are inherently structural and hit
  dedicated paths; the free-form REPL is the noise. Persist canvas
  edits always; gate REPL evals through A/B (or treat REPL as
  scratch). Matches "canvas = the topology I built; REPL =
  scratchpad". Needs an origin flag on the eval endpoint.
- *F — form-splitting.* Split a multi-form eval into top-level forms,
  eval each, persist only the forms that moved the structural signal.
  Fixes the mixed-form hole; more work. A refinement on B.
- *G — different model: regenerate, don't append.* Stop append-logging
  evals; serialize the live graph structure to the file (the
  clipboard/paste code already emits categories + internal edges, so
  there's precedent). The file becomes a canonical snapshot, never
  holds transient pokes, no replay surprises. Cons: bigger change;
  regenerated forms lose exact source spelling (use defaults, like
  paste) and you'd need a separate channel for non-graph config
  (defaults/knobs) you still want kept.

*** Recommendation

B as the gate, with the policy widened to "structural OR
config-worth-keeping": a structural-change counter gates =make-*= /
=connect= / =disconnect= / =remove= (and =rename= if names should
persist), plus a small explicit allowlist for non-graph config evals
(defaults =setq=s) so they still persist. Optionally add D so the
REPL is scratch-by-default. Minimal, lives at the five mutators, and
fixes the reload-replay-of-pokes bug as a bonus.

Open decisions before speccing:
1. Defaults edits — keep persisting (likely yes) or drop as not-graph?
2. Renames — graph edit (persist) or not?
3. REPL vs canvas — REPL scratch-by-default (D), or persist whenever
   the graph changed regardless of origin?

** DONE Undo in the topology editing canvas

Cut / copy / paste land via =/api/mg/{id}/eval= and update the
override file. There's no undo for the canvas-level operations
(delete, paste, connect, disconnect, rename). The persisted-
overrides dialog can hand-prune individual forms after the fact,
but it's a row-by-row checkbox UI, not a Ctrl-Z stack.

Want a per-mg undo stack the canvas + side-panel mutations push
onto: each entry is enough state to revert. Two shapes:

- *Optimistic snapshots.* Snapshot the override file before each
  eval; Ctrl-Z restores it. Cheap but heavy (whole-file write per
  edit) and doesn't help if the user evaluates via REPL.
- *Inverse operations.* Each mutating verb produces an inverse
  expression — =(rename-component 1 "old")=, =(connect a b)= →
  =(disconnect a b)=, etc. — that the stack pushes. Lighter but
  needs every mutating defun to declare its inverse.

Probably start with snapshots (simpler) and refine.

Acceptance: Ctrl-Z after a paste removes the pasted subgraph;
Ctrl-Z after a rename restores the previous name; Ctrl-Shift-Z
re-applies the undone step.
