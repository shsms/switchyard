#+TITLE: Switchyard — UI redesign proposal
#+TODO: TODO IN-PROGRESS WAIT | DONE CANCELLED

Notes + proposed work items to shift the SPA from "topology canvas
that incidentally shows data" toward "live dashboard that
incidentally hosts a topology canvas". Goal is convenience for a
developer who is soaking their downstream control app against the
simulator over a long session — they need at-a-glance state, easy
scenario picks, easy fault injection, and the topology only when
they're editing structure.

* Targets

** Display

Primary target is a 32" 4K monitor (3840 × 2160 at 100 % scale or
1920 × 1080 effective at 200 %). Plenty of horizontal real estate
— design for *side-by-side density* rather than stacked-column
fallbacks. Dashboard tier rows should fan out into multi-column
grids on wide displays:

  - Tier 2 (battery pool + individuals): pool row on the left,
    per-battery rows in a 2 — 3 column grid on the right.
  - Tier 3 (PV pool + inverters): same shape.
  - Tier 5 (EV / CHP): two columns side-by-side rather than
    stacked.

Keep a single-column fallback for sub-1400 px viewports so a
laptop / iPad screen still works, but optimise the default layout
for the wide case. Mirror tradingsim's =.tier-row= +
=minmax(0, Nfr)= grid pattern (= src/ui-assets/style.css =76-87).

Compact-density mode (A3) further densifies the wide layout —
useful for showing every battery, every inverter, the pulse bar,
and the topology canvas's mini-map all at once without scrolling.

** Browser testing

The dev environment has Selenium + Firefox ESR + Chromium with
headless support — same stack tradingsim's =tests/ui_selenium.py=
uses (see =/vagrant/tradingsim/CLAUDE.md=, "Browser tests"
section). =geckodriver= and =chromedriver= are both installed; the
system =python3-selenium= package is at =/usr/lib/python3/dist-
packages= (call via =/usr/bin/python3= directly, not the project
venv).

Use this stack for browser tests of the Dashboard, the Scenarios
mode, and any interaction that depends on layout / CSS / JS
behaviour the existing axum =oneshot= integration tests can't
exercise. Drop Playwright from =todo.org='s F-series in favour of
the Selenium path — fewer Node / npm dependencies, same coverage.

Test harness shape (matches the tradingsim pattern):

  - Boot switchyard on a free port via a pytest / Python fixture.
  - Open the page in headless Firefox.
  - Assert against DOM structure (=By.CSS_SELECTOR=), not
    pixels — topology layout is non-deterministic across runs.
  - For Dashboard tiles: assert the number renders within a
    bounded staleness after a known input (e.g. =set-meter-power=
    via =/api/eval=, wait for the tile's text content to update).

Inspired by a side-by-side analysis with the tradingsim SPA
(/vagrant/tradingsim/ui-assets/) — the patterns called out under
each item below either originated there or are the
microgrid-shaped analogue of something there.

Also informed by the two Frequenz Rust crates that downstream apps
are most likely to be written against:

  - frequenz-microgrid-rs: =Microgrid::try_new(url, config)= →
    typed metric streams via =LogicalMeterHandle::grid /battery
    /pv /chp /ev_charger /consumer /producer<M>=, parameterised
    over =metric::AcPowerActive= / =SoC= / =Voltage= / etc. Plus
    =BatteryPool= as a first-class aggregation (combined bounds +
    SoC). Quantities (=Power=, =Current=, =Voltage=, =Frequency=)
    are typed and unit-checked. The surface is *named*
    aggregations only — =grid<M>()=, =battery<M>(Option<ids>)=,
    =pv<M>(Option<ids>)=, =chp/ev_charger<M>(Option<ids>)=,
    =consumer<M>()=, =producer<M>()=, =component<M>(id)= — plus
    the public composable =Formula= enum
    (=Add/Subtract/Coalesce/Min/Max/Avg/…=) for hand-built ASTs.
    There is no =Formula::from_str= parser; the graph crate's
    rendered formula strings are inspection output, not evaluator
    input.
  - frequenz-microgrid-component-graph-rs: graph crate underneath.
    DAG validation against per-category neighbour rules, formula
    string generation (=#1 + COALESCE(#2, #3, 0.0)=), pass-through
    handling for transformers/breakers/converters/HVAC etc.

The integration shape: switchyard's UI server runs
=frequenz-microgrid= as a client pointed at switchyard's own gRPC
server on localhost — switchyard is both Microgrid server and
Microgrid client, of itself. Aggregated Dashboard tiles (grid,
battery pool, PV pool, consumer / producer aggregates) come from
=LogicalMeterHandle::xxx<Q>().subscribe()= the same way a
production control app would consume them. Per-component charts
keep flowing from switchyard's physics history rings via the
existing =/api/history=. This means the Dashboard's pool numbers
match what the developer's app sees not because switchyard
reimplemented the formulas, but because both clients use the same
library.

* Integration

** DONE Z1. Adopt frequenz-microgrid-component-graph for validation

Implement =Node= + =Edge= on switchyard's existing component +
connection types in =src/sim/component.rs= /
=src/sim/world.rs=. Use =ComponentGraph::try_new= at the end of
each config-load / hot-reload cycle. Three immediate payoffs:

- Per-category neighbour validation (battery has battery-inverter
  predecessor, etc.) replaces the ad-hoc checks scattered across
  =%make-*= defuns. Error messages become canonical.
- Pass-through handling (transformers, breakers, converters,
  HVAC, electrolyzers, capacitor banks) is free. Switchyard
  doesn't model these yet but the moment it does, formulas and
  the canvas's "logical view" stay correct.
- Single source of truth for the topology validator: switchyard
  and any downstream app that uses the graph crate hit the same
  validation rules. A scenario that wires up an invalid topology
  via the lisp REPL fails in switchyard the same way it would
  fail in a production deployment.

Surface validation outcome in the chrome:
- Errors → red toast + canvas freeze (current behaviour for lisp
  errors, but with the graph crate's message text).
- Warnings (=allow_*= flags downgraded) → yellow pill in the
  pulse bar that opens a dialog listing the warnings.

Acceptance:
- =ComponentGraph::try_new= runs on every reload; failure prevents
  the topology from being swapped in.
- The graph's =predecessors= / =successors= iterators are used by
  the inspector pane (replacing the current hand-rolled lookup).

** DONE Z2. Drive Dashboard aggregates through frequenz-microgrid's
   own LogicalMeterHandle

Switchyard's UI server becomes a =frequenz-microgrid= client of
the switchyard gRPC server it's running alongside. The Dashboard
doesn't evaluate formulas itself — it subscribes to the same
typed streams a downstream control app would:

  // in the UI server task, once per boot
  let microgrid = Microgrid::try_new(
      grpc_addr,
      LogicalMeterConfig::new(TimeDelta::seconds(1)),
  ).await?;
  let grid_power     = microgrid.logical_meter().grid::<AcPowerActive>()?.subscribe().await?;
  let battery_pool_p = microgrid.battery_pool().active_power().subscribe().await?;
  let pv_power       = microgrid.logical_meter().pv::<AcPowerActive>()?.subscribe().await?;
  // …per Dashboard tier

Each subscription is a typed =Sample<Q>= stream. The UI server
fans these out over the existing =/ws/events= channel (one frame
shape per metric, carrying =unit= + =value=). The SPA's tier
panels bind directly.

Two payoffs over a hand-rolled evaluator:

1. The graph-crate formulas, =BatteryPool= aggregation,
   =combine_parallel= bounds math, =COALESCE= for missing
   meters, =prefer_meters_in_component_formulas=, per-formula
   overrides — switchyard gets all of them for free because
   =frequenz-microgrid= already does this work.
2. The Dashboard's numbers are computed by the *same code path*
   a production app would use. A developer comparing
   "switchyard says 18.2 kW pool, my app says 17.6 kW" is now
   doing a stream-shape / config diff against the same client
   library, not against a switchyard reimplementation that
   could drift.

The =Option<BTreeSet<u64>>= "ids" parameter on
=battery/pv/chp/ev_charger= scopes the aggregation to a subset of
components — useful for "show the south-yard battery pool"
without writing a custom formula. Truly ad-hoc aggregations
(e.g. one specific branch) would compose the public =Formula=
enum by hand; for the Dashboard's planned tiers, named
aggregations + scope-by-ids covers everything.

Note on layering: switchyard ends up being both Microgrid server
and Microgrid client (of itself, on localhost). The overhead is
trivial (loopback gRPC at 1 Hz) and the consistency win is the
whole point — if =frequenz-microgrid='s stream is broken,
switchyard's own dashboard surfaces it.

Alternative considered: =MicrogridClientHandle::new_from_client=
takes an =impl MicrogridApiClient= trait, so the dashboard could
in principle skip the network entirely by impl'ing the trait
against =&World= and handing the handle to
=Microgrid::new_from_handles=. The trait surface is four async
methods plus a =TelemetryStream= associated type. Rejected for
this round:

- ~200 LOC of in-process proto conversion + stream adapter vs
  ~5 LOC of =Microgrid::try_new(url, …).await=.
- Duplicates switchyard's existing server-side conversion in
  =proto_conv.rs= / =server.rs= — two implementations of the
  same gRPC contract, with drift risk.
- The loopback path exercises switchyard's own gRPC server on
  every dashboard request — a server-side regression breaks the
  dashboard loudly. The in-process path would mask that.

Performance is a non-factor: localhost gRPC at the UI's 1 Hz
cadence is ~1 ms per RPC.

Phased rollout:
- The existing =/api/history?metric=...&id=...= for per-component
  charts stays, fed by the physics history rings.
- The new aggregated tiles (battery pool, PV pool, grid, consumer,
  producer) all read from =frequenz-microgrid= subscriptions.

Acceptance:
- Dashboard's pool tiles render numbers that match what =swctl
  pool battery --stream= prints (both go through
  =frequenz-microgrid=, so this is a tautology — but it's the
  invariant we want).
- Dashboard hover-tooltip on a pool number shows the formula
  string that produced it (asked of =ComponentGraph= directly;
  =frequenz-microgrid= exposes the underlying graph).

** DONE Z3. Typed-quantity payloads on the WS / history channels

=frequenz-microgrid='s subscriptions yield =Sample<Q>= where
=Q= is =Power= / =ReactivePower= / =Voltage= / =Frequency= /
=Percentage= / =Energy= / =Current=, each with an explicit
=base_unit()=. Use that as the wire shape for the new aggregated
streams (Z2) and retrofit the existing per-component history
endpoint:

  /api/history?metric=active_power
    → { quantity: "Power", unit: "W", samples: [{ ts, value }] }

  /api/history?metric=soc
    → { quantity: "Percentage", unit: "%", samples: [{ ts, value }] }

The WS aggregated-stream frames carry the same =quantity= +
=unit= fields. UI's chart auto-scale (W → kW → MW) already
exists; this just removes the JS-side metric-name parsing.

Acceptance:
- Chart axis label updates when toggling between =active_power=
  and =reactive_power= because the response =unit= differs.
- Two metric streams of the same quantity (e.g. battery-pool P
  and grid P) auto-scale together within a tier so the rows
  share a visual baseline.

** DONE Z4. BatteryPool / PvPool as first-class Dashboard rows
   - Battery pool envelope (lower + upper) wired through
     ~BatteryPool::power_bounds~. Pool SoC and PvPool aren't on
     frequenz-microgrid 0.4.1's surface yet — revisit on upgrade.

=Microgrid::battery_pool()= returns a =BatteryPool= with
aggregated active-power, =Bounds<Power>=, and =SoC= subscriptions
ready to go. The Dashboard's tier-2 (batteries) gains a "pool"
row at the top fed by those subscriptions; tier-3 (PV) gets the
PV equivalent; tier-4 leans on the graph crate's
=consumer_formula= and =producer_formula= via the logical-meter
handle.

This is the row a control-app developer cares about most — their
SetPower call targets the pool, not an individual battery, and
they want to see the aggregated answer.

The =combine_parallel= bounds math, =COALESCE= over missing
component samples, and =prefer_meters_in_component_formulas=
selection all live inside =frequenz-microgrid= — switchyard just
subscribes.

Acceptance:
- Sum-of-individuals = pool row, within tick-tolerance.
- A fault that errors one battery shows the pool's bounds
  shrinking by that battery's contribution (the pool's
  =Bounds<Power>= stream emits the new envelope automatically).
- The Dashboard renders the pool row even before all individual
  batteries have reported their first sample, because the pool
  stream =COALESCE='s missing values.

** WAIT Z5. Import topology from frequenz-microgrid's

A developer who already targets a specific deployment has a

  swctl import-topology path/to/site-graph.{yaml,toml}
  POST /api/topology/import   (multipart body)

…which produces a config.lisp / overrides sidecar that recreates

category defaults from =sim/defaults.lisp= fill in physics knobs
the production graph doesn't carry (rated power, ramp, SoC).

Cuts the "build a switchyard topology that matches my deployment"
step from a manual translation to one command.

Acceptance:
- A site-graph file with one grid + main meter + 2 batteries +
  1 PV reproduces in switchyard with all ids matching.
- A subsequent =swctl tree= shows the same shape =frequenz-
  microgrid='s =ComponentGraph= would.

** DONE Z6. Surface graph-crate warnings on the canvas

The graph crate emits =tracing::warn!= per pass-through node and

(simple subscriber) and:

- Pass-through nodes render dashed-edged on the canvas (current
  hidden-meter cue is already in this shape — extend to the

** TODO Z7. Open question — ad-hoc / custom aggregations

=LogicalMeterHandle='s surface is named aggregations only.
Scope-by-ids on =battery/pv/chp/ev_charger= covers "pool over a
subset"; everything else (arbitrary branch sum, "every component
under this meter", custom mix) needs either:

a. Hand-built =Formula= AST via the public enum's variants
   (=Add/Subtract/Coalesce/…=). Doable; not ergonomic.
b. A switchyard-side helper that walks the graph crate's
   =predecessors=/=successors= and emits a =Formula= AST for a
   subtree. Would live in switchyard, not upstream.
c. Skip aggregation: surface the participating components as a
   row group on the Dashboard, let the user eyeball the sum.

Worth flagging because the topology canvas's "select N components"
gesture is a natural launching point — "sum these into a custom
tile" would feel like a feature, but the path through
=frequenz-microgrid= isn't paved yet. Defer until a concrete use
case lands.

* Layout

** DONE A1. Promote the topology to one of three top-level modes

Today the canvas is "the UI". For long soak runs it's mostly
noise — the developer cares what the components are *doing*,
not where they sit on a graph. Three modes the chrome toggles
between:

  [Dashboard] [Topology] [Scenarios]

Each mode keeps the REPL drawer + the inspector side panel.
Only the main pane changes. Default mode = Dashboard;

Acceptance:
- Three top-bar buttons toggle the visible main pane.
- Selection still updates the inspector regardless of mode.
- Page refresh restores the last-picked mode.

* Scenarios

** DONE B1. First-class scenario registry (server + lisp DSL)

Today a scenario is a =(load …)=-d script with timer chains.
Replace with a registered entity per tradingsim's
=ScenarioDef=:

  (define-scenario
    :name "evening-peak"
    :description "Consumer ramp 17:00–21:00, PV gone, batteries discharging"
    :date "2026-01-15"  ;; optional: pins curves to a calendar day
    :stages
    '((:name "ramp"    :hour-from 17 :hour-to 18
       :overrides ((:consumer-power-fn (lambda () (linterp 5000 25000 0 1))))
       :weather (:cloud-cover 0.7 :temperature-base 273))
      (:name "peak"    :hour-from 18 :hour-to 20
       :overrides ((:consumer-power 25000)))
      (:name "wind-down" :hour-from 20 :hour-to 21
       :overrides ((:consumer-power-fn (lambda () (linterp 25000 6000 0 1)))))))

Server-side =SharedScenarios= registry, runtime-state per
entry (=current_stage=, =started_at=, =manual_override=). Stage
transition writes the override set onto the relevant component
shared configs; reverting drops back to baseline.

Acceptance:
- =(define-scenario)= registers; survives hot reload (Arc-
  identity invariant, like FleetParams in tradingsim).
- A running scenario's stage auto-advances by wallclock unless
  =manual_override= is set (operator clicked Prev/Next).

** DONE B3. HTTP API parity with tradingsim's scenarios

Endpoints:
  GET    /api/scenarios                         registry list
  POST   /api/scenarios/{name}/start            start fresh
  POST   /api/scenarios/{name}/next             advance one stage
  POST   /api/scenarios/{name}/prev             back one stage

The existing /api/scenario/report + /api/scenario/events

  swctl scenarios start evening-peak

  swctl scenarios report --tail   (streams journal events)

Acceptance: round-trip via curl + via swctl.

** DONE B4. Ship a starter scenario library

Pre-registered out of the box, picked from the UI dropdown
without writing lisp. Each is two-to-four stages:

  - peak-evening-load         consumer ramp 17–21, PV off
  - pv-dropout                clear → cloud → clear
  - battery-degraded-fleet    one battery in error half the time
  - flaky-network             solar inverter silent + commands timeout
  - frequency-deviation       grid frequency drifts ±100 mHz
  - cold-start                 every component starts in 'standby
  - off-grid-island           grid =error=, battery & PV only

The last two are particularly useful with Z4's BatteryPool: a
control-app developer can verify their state-machine handles
=pool.bounds.upper = 0= (every battery sidelined or off-grid).

Lives in =scenarios/library/= and loaded by config.lisp.

Acceptance: dropdown shows seven entries on a fresh checkout.

* Fault injection

** TODO E1. Fault-injection side-panel mode

A side-panel mode (alongside Inspect / Defaults / Report)
listing one-click named faults. Target = current selection
(or a category if no selection):

  Fault injection                target: battery 1003
  [ Mark 'error 60s ]   [ Flaky network 5 min ]
  [ Telemetry silent 30s ]   [ Command timeout 60s ]
  [ Stuck at 0 W ]   [ Bounds clamp ±1 kW for 5 min ]

adds a journal event so the Report panel records the action;
when the timer reverts, another event lands.

Acceptance:
- Clicking a chip with a battery selected immediately changes
  its state + appends a journal event.
- Reverting fires automatically; the chip stops looking
  "armed".
- The pool-targeting variants iterate the graph crate's
  =battery_predecessors= / similar accessors to find the right
  set of components.
