#+TITLE: Switchyard — UI redesign proposal
#+TODO: TODO IN-PROGRESS | DONE CANCELLED

Notes + proposed work items to shift the SPA from "topology canvas
that incidentally shows data" toward "live dashboard that
incidentally hosts a topology canvas". Goal is convenience for a
developer who is soaking their downstream control app against the
simulator over a long session — they need at-a-glance state, easy
scenario picks, easy fault injection, and the topology only when
they're editing structure.

Inspired by a side-by-side analysis with the tradingsim SPA
(/vagrant/tradingsim/ui-assets/) — the patterns called out under
each item below either originated there or are the
microgrid-shaped analogue of something there.

Also informed by the two Frequenz Rust crates that downstream apps
are most likely to be written against:

  - frequenz-microgrid-rs: =Microgrid::try_new(url, config)= →
    typed metric streams via =LogicalMeterHandle::grid /battery
    /pv /chp /ev_charger /consumer /producer<M>=, parameterised
    over =metric::AcPowerActive= / =SoC= / =Voltage= / etc. Plus
    =BatteryPool= as a first-class aggregation (combined bounds +
    SoC). Quantities (=Power=, =Current=, =Voltage=, =Frequency=)
    are typed and unit-checked.
  - frequenz-microgrid-component-graph-rs: graph crate underneath.
    DAG validation against per-category neighbour rules, formula
    string generation (=#1 + COALESCE(#2, #3, 0.0)=), pass-through
    handling for transformers/breakers/converters/HVAC etc.

These crates are what developers consuming the production
Microgrid API write their apps against. Aligning switchyard's UI
data shapes with them means a dev debugging "my app reads the
wrong battery-pool SoC" can compare what switchyard's Dashboard
shows for the same =battery_formula= to what their client computes,
side-by-side. That's the convenience target.

* Integration

** TODO Z1. Adopt frequenz-microgrid-component-graph for validation

Implement =Node= + =Edge= on switchyard's existing component +
connection types in =src/sim/component.rs= /
=src/sim/world.rs=. Use =ComponentGraph::try_new= at the end of
each config-load / hot-reload cycle. Three immediate payoffs:

- Per-category neighbour validation (battery has battery-inverter
  predecessor, etc.) replaces the ad-hoc checks scattered across
  =%make-*= defuns. Error messages become canonical.
- Pass-through handling (transformers, breakers, converters,
  HVAC, electrolyzers, capacitor banks) is free. Switchyard
  doesn't model these yet but the moment it does, formulas and
  the canvas's "logical view" stay correct.
- Single source of truth for the topology validator: switchyard
  and any downstream app that uses the graph crate hit the same
  validation rules. A scenario that wires up an invalid topology
  via the lisp REPL fails in switchyard the same way it would
  fail in a production deployment.

Surface validation outcome in the chrome:
- Errors → red toast + canvas freeze (current behaviour for lisp
  errors, but with the graph crate's message text).
- Warnings (=allow_*= flags downgraded) → yellow pill in the
  pulse bar that opens a dialog listing the warnings.

Acceptance:
- =ComponentGraph::try_new= runs on every reload; failure prevents
  the topology from being swapped in.
- The graph's =predecessors= / =successors= iterators are used by
  the inspector pane (replacing the current hand-rolled lookup).

** TODO Z2. Adopt component-graph formulas for the Dashboard's
   aggregated tiles

Today the scenario reporter hand-codes "sum across batteries",
"sum across PV". The graph crate emits formulas:

  battery_formula  → "#1003 + #1006 + COALESCE(…)"
  pv_formula       → "#200"
  grid_formula     → "#2"
  consumer_formula → …
  producer_formula → "#200 + #1003 + …"

Implement a tiny evaluator (or borrow one — the formulas are
trivial arithmetic over component-id references) and feed the
Dashboard's aggregated rows. Same formulas the production
=Microgrid= client computes against, so the numbers match.

Acceptance:
- Dashboard's "battery pool" row shows the result of the literal
  =battery_formula= string. The string itself is visible in a
  tooltip / debug overlay so a developer reading "20 kW" can see
  it came from =#1003 + COALESCE(#1006, 0)=.

** TODO Z3. Typed-quantity metric streams on /api/history

frequenz-microgrid exposes =Sample<Q>= carrying =timestamp() +
value(): Q= where =Q= is =Power= / =Voltage= / =SoC= / etc. Each
quantity has explicit unit conversions.

Today switchyard's =/api/history?metric=active_power_w= returns
raw floats with the unit baked into the name. Restructure:

  /api/history?metric=active_power
    → { unit: "W", samples: [{ ts, value }] }

  /api/history?metric=soc
    → { unit: "%", samples: [{ ts, value }] }

  /api/history?metric=frequency
    → { unit: "Hz", samples: [{ ts, value }] }

UI renders the unit from the response, applies the same auto-
scale (W → kW → MW) it already does for power. Lets the chart
title carry the correct unit without the JS guessing from the
metric name.

Acceptance: chart axis label updates when the toggle between
=active_power= and =reactive_power= flips the unit.

** TODO Z4. BatteryPool / PvPool / ConsumerPool as first-class
   Dashboard rows

frequenz-microgrid surfaces =BatteryPool= with =Bounds<Power>= and
=Percentage= SoC aggregates. The Dashboard's tier-2 (batteries)
gains a "pool" row at the top, computed via the graph crate's
=battery_formula= for instantaneous power and =Bounds::combine_
parallel= over each battery's current bounds for the envelope.
Same shape for tier-3 (PV) and a new tier-4 (consumers).

This is the row a control-app developer cares about most — their
SetPower call targets the pool, not an individual battery, and
they want to see the aggregated answer.

Acceptance:
- Sum-of-individuals = pool row, within tick-tolerance.
- Pool bounds = =combine_parallel= over the per-battery bounds.
- A fault that errors one battery shows the pool's bounds
  shrinking by that battery's contribution.

** TODO Z5. Import topology from frequenz-microgrid's
   ComponentGraphConfig format

A developer who already targets a specific deployment has a
config — typically YAML or proto-text — that the production
graph crate consumes. Add a switchyard import path:

  swctl import-topology path/to/site-graph.{yaml,toml}
  POST /api/topology/import   (multipart body)

…which produces a config.lisp / overrides sidecar that recreates
the same graph in switchyard. Component ids preserved. Per-
category defaults from =sim/defaults.lisp= fill in physics knobs
the production graph doesn't carry (rated power, ramp, SoC).

Cuts the "build a switchyard topology that matches my deployment"
step from a manual translation to one command.

Acceptance:
- A site-graph file with one grid + main meter + 2 batteries +
  1 PV reproduces in switchyard with all ids matching.
- A subsequent =swctl tree= shows the same shape =frequenz-
  microgrid='s =ComponentGraph= would.

** TODO Z6. Surface graph-crate warnings on the canvas

The graph crate emits =tracing::warn!= per pass-through node and
per =allow_*=-downgraded validation failure. Capture these
(simple subscriber) and:

- Pass-through nodes render dashed-edged on the canvas (current
  hidden-meter cue is already in this shape — extend to the
  pass-through case).
- A "graph warnings" pill in the pulse bar shows the count and
  opens a dialog with the full list.

Acceptance: a config that registers an unknown pass-through
category shows the pill within one reload cycle.

* Layout

** TODO A1. Promote the topology to one of three top-level modes

Today the canvas is "the UI". For long soak runs it's mostly
noise — the developer cares what the components are *doing*,
not where they sit on a graph. Three modes the chrome toggles
between:

  [Dashboard] [Topology] [Scenarios]

Each mode keeps the REPL drawer + the inspector side panel.
Only the main pane changes. Default mode = Dashboard;
preference persists in localStorage (mirrors the dropdown-
choice persistence pattern in tradingsim's app.js).

Acceptance:
- Three top-bar buttons toggle the visible main pane.
- Selection still updates the inspector regardless of mode.
- Page refresh restores the last-picked mode.

** TODO A2. Dashboard view — tiered panels analogous to tradingsim

Stacked tiles, each reading from existing /api/topology +
/api/history + the formulas from Z2. Each tile collapses cleanly
when no components of that category are registered.

  +-----------------------------------------------------------+
  |  PULSE BAR  (see C1)                                      |
  +-----------------------------------------------------------+
  |  Tier 1 (full width):  Grid connection point              |
  |    big number (current P) · current frequency · voltage   |
  |    30-min sparkline · =grid_formula= debug tooltip        |
  +-----------------------------------------------------------+
  |  Tier 2:  Battery pool + per-battery (see Z4)             |
  |    pool row: P · bounds envelope · pool-SoC               |
  |    then row per battery: name, SoC bar, DC W, bounds,     |
  |    health, 30 s sparkline                                 |
  +-----------------------------------------------------------+
  |  Tier 3:  PV pool + per-inverter                          |
  |    pool row: P · =pv_formula= · =pv_ac_coalesce_formula=  |
  |    rows: setpoint W, measured W, ±bounds, lifetime, spark |
  +-----------------------------------------------------------+
  |  Tier 4:  Consumer / Producer aggregates                  |
  |    =consumer_formula= + =producer_formula= summary cards  |
  +-----------------------------------------------------------+
  |  Tier 5:  EV / CHP — one row each                         |
  +-----------------------------------------------------------+

The "pool row first, individuals second" structure makes the
dashboard read top-to-bottom in increasing detail — the developer
sees the aggregate their app commands, then drills into which
individual is dragging it.

Acceptance:
- All data comes from current endpoints + Z1/Z2 formulas — no
  new server work beyond the integration items.
- Rows highlight when a value pins to a bound or when health
  ≠ =ok=.

** TODO A3. Density toggle

CSS-only switch in the chrome that shrinks panel padding +
fonts. Useful when the dashboard's tier grid is dense.
localStorage-persisted, mirrors tradingsim's pattern
(=#density-toggle=).

Acceptance: one click toggles a =.compact= class on =<body>=;
all panel paddings, font-sizes, and sparkline heights respond.

* Pulse bar

** TODO C1. Always-on system pulse strip

Persistent strip below the header, visible in every mode (not
just Dashboard). Mirrors tradingsim's =.pulse= section, scaled
to microgrid concerns:

  | setpoints ▇▆▆▇▃ ▆▆▇▇▇  | OK 7 · STDBY 1 · ERR 0 | graph: ✓ | scenario: flaky-network ▶ 2/4 | 14:23:11 CEST [local] [compact] |

Components:
- Setpoint-rate sparkbar (per-second buckets over last 60 s)
  fed off the existing /ws/events =setpoint= frames.
- Health pill — three counters from /api/topology, click to
  filter the Dashboard to that bucket.
- Graph pill (from Z6) — ✓ / ⚠ N warnings; clickable.
- Scenario chip — name + current stage / total, click to jump
  to Scenarios mode.
- Clock + TZ toggle (UTC ↔ configured zone — see D1).
- Density chip (see A3).

Acceptance:
- Updates without polling — pulse counters tick off the
  existing WS stream.
- Health-pill click scrolls Dashboard to the matching tier.

* Scenarios

** TODO B1. First-class scenario registry (server + lisp DSL)

Today a scenario is a =(load …)=-d script with timer chains.
Replace with a registered entity per tradingsim's
=ScenarioDef=:

  (define-scenario
    :name "evening-peak"
    :description "Consumer ramp 17:00–21:00, PV gone, batteries discharging"
    :date "2026-01-15"  ;; optional: pins curves to a calendar day
    :stages
    '((:name "ramp"    :hour-from 17 :hour-to 18
       :overrides ((:consumer-power-fn (lambda () (linterp 5000 25000 0 1))))
       :weather (:cloud-cover 0.7 :temperature-base 273))
      (:name "peak"    :hour-from 18 :hour-to 20
       :overrides ((:consumer-power 25000)))
      (:name "wind-down" :hour-from 20 :hour-to 21
       :overrides ((:consumer-power-fn (lambda () (linterp 25000 6000 0 1)))))))

Server-side =SharedScenarios= registry, runtime-state per
entry (=current_stage=, =started_at=, =manual_override=). Stage
transition writes the override set onto the relevant component
shared configs; reverting drops back to baseline.

Acceptance:
- =(define-scenario)= registers; survives hot reload (Arc-
  identity invariant, like FleetParams in tradingsim).
- A running scenario's stage auto-advances by wallclock unless
  =manual_override= is set (operator clicked Prev/Next).

** TODO B2. Scenarios mode (UI panel)

Mirrors =renderScenarioActive= in tradingsim's app.js:
- Day-long horizontal timeline strip (00:00 → 24:00), one
  block per stage proportional to its hour span.
- "Now" marker dropped on the timeline at the current
  local-hour.
- Click any block → jump to that stage (manual-override on).
- Below the strip: row-per-stage list with name, time,
  bias-or-override summary, weather chips.
- Above: prev/next/stop buttons + manual-override badge.

Acceptance:
- Renders the registered scenarios at mode entry.
- prev / next / jump POST to /api/scenarios/.../jump/N and
  reflect within the next poll tick.

** TODO B3. HTTP API parity with tradingsim's scenarios

Endpoints:
  GET    /api/scenarios                         registry list
  POST   /api/scenarios/{name}/start            start fresh
  POST   /api/scenarios/{name}/next             advance one stage
  POST   /api/scenarios/{name}/prev             back one stage
  POST   /api/scenarios/{name}/jump/{idx}       jump
  POST   /api/scenarios/{name}/stop             freeze + reset

The existing /api/scenario/report + /api/scenario/events
endpoints keep their shape — the report is the aggregate, the
new endpoints are control. Mirror in =swctl=:

  swctl scenarios list
  swctl scenarios start evening-peak
  swctl scenarios next | prev | jump N | stop
  swctl scenarios report --tail   (streams journal events)

Acceptance: round-trip via curl + via swctl.

** TODO B4. Ship a starter scenario library

Pre-registered out of the box, picked from the UI dropdown
without writing lisp. Each is two-to-four stages:

  - peak-evening-load         consumer ramp 17–21, PV off
  - pv-dropout                clear → cloud → clear
  - battery-degraded-fleet    one battery in error half the time
  - flaky-network             solar inverter silent + commands timeout
  - frequency-deviation       grid frequency drifts ±100 mHz
  - cold-start                 every component starts in 'standby
  - off-grid-island           grid =error=, battery & PV only

The last two are particularly useful with Z4's BatteryPool: a
control-app developer can verify their state-machine handles
=pool.bounds.upper = 0= (every battery sidelined or off-grid).

Lives in =scenarios/library/= and loaded by config.lisp.

Acceptance: dropdown shows seven entries on a fresh checkout.

** TODO B5. Scenario hooks for the pool-level abstractions

Scenarios that target a category instead of individual
components should write through the pool shape:

  (scenario-event 'pool-bounds-shrink "battery 1003 sidelined")

And the Dashboard's pool row should react: =combine_parallel=
over the surviving batteries' bounds, refresh the pool envelope
visual. This is what the graph crate's pass-through-aware
=battery_formula= already produces; Z2 wires it up, B5 ensures
scenarios journal the *reason* alongside the math.

Acceptance: a scenario that errors a battery emits a journal
event and the Dashboard's pool envelope visibly contracts.

* Wallclock + sim-time

** TODO D1. Wallclock-anchored Clock primitive

Today curves are =(window-elapsed 900.0)= — anchored to the
binary's start, not the time of day. Add =Clock=:

  (set-timezone "Europe/Berlin")     ;; defaults to Europe/Berlin
  (local-hour)                       ;; fractional hour, 0..24
  (local-day-of-year)
  (local-date)

Wire into:
- Scenario stage auto-advance (B1).
- The pulse-bar clock display + TZ toggle (C1).
- A =tz= field on /api/topology that the UI uses to format
  per-component timestamps.

Acceptance:
- Reloading config.lisp does not reset the clock.
- UI's TZ toggle flips clock + any per-component timestamps
  the inspector renders.

** TODO D2. Sim-time multiplier (deferred, tradingsim has same TODO)

=(set-sim-time-multiplier 60)= → one wallclock second = one sim
minute. Scenarios accept their hour stages in sim time.
=(every-sim …)= sugar layers on top of =every=.

Useful for soaking an app against a 24 h scenario in 24 min.

Acceptance: a 24h scenario completes in real wallclock time
≈ 24h / multiplier; sample timestamps are in sim time, not
wallclock.

* Fault injection

** TODO E1. Fault-injection side-panel mode

A side-panel mode (alongside Inspect / Defaults / Report)
listing one-click named faults. Target = current selection
(or a category if no selection):

  Fault injection                target: battery 1003
  [ Mark 'error 60s ]   [ Flaky network 5 min ]
  [ Telemetry silent 30s ]   [ Command timeout 60s ]
  [ Stuck at 0 W ]   [ Bounds clamp ±1 kW for 5 min ]
  [ SoC drift +10% over 10 min ]   [ Custom… ]

Plus pool-targeting variants (lean on Z4):

  [ Pool: take 50% offline ]   [ Pool: clamp to 0 W ]

Each chip = a small lisp expression behind the scenes. Firing
adds a journal event so the Report panel records the action;
when the timer reverts, another event lands.

Acceptance:
- Clicking a chip with a battery selected immediately changes
  its state + appends a journal event.
- Reverting fires automatically; the chip stops looking
  "armed".
- The pool-targeting variants iterate the graph crate's
  =battery_predecessors= / similar accessors to find the right
  set of components.

** TODO E2. CLI parity for fault injection

  swctl fault flaky-network 1003 --duration 5m
  swctl fault telemetry-silent 200 --duration 30s
  swctl fault mark-error 1003 --duration 60s
  swctl fault custom 'some-lisp-expr' --revert 'other-expr' --duration 1m
  swctl fault pool battery offline --pct 50 --duration 5m

Mirrors E1. The chips and the CLI share one shared registry of
named fault recipes (see E3).

** TODO E3. Fault registry — =(define-fault NAME …)=

Defuns shared between UI and CLI. Lisp configuration declares
named faults the same way =(define-scenario)= does:

  (define-fault
    :name "flaky-network"
    :description "telemetry silent + commands time out"
    :on (lambda (id) (set-component-telemetry-mode id 'silent)
                     (set-component-command-mode id 'timeout))
    :off (lambda (id) (set-component-telemetry-mode id 'normal)
                      (set-component-command-mode id 'normal)))

The UI's chip list reads from this registry; new faults appear
without UI changes. Ship a library of the chips listed in E1.

** TODO F3. Setpoint envelope visual

Generalised renderer for =(lower, current, upper)= triples.
Used in F1 (SoC bar 0..100) and F2 (active-P envelope from
=Bounds<Power>=). Single uPlot recipe so the visual vocabulary
is consistent.

Hook to Z3's typed quantities: the envelope renderer takes a
=Quantity= type, picks units, and renders consistently
whether the underlying value is Power, Current, or Percentage.

** TODO F4. Formula-debug overlay

Hover any aggregated number on the Dashboard → tooltip shows
the formula string emitted by the graph crate (Z2). Click → a
side-panel mode lists the formula tree with each =#N= node
linked to the topology canvas. Lets a developer trace "why
does =consumer_formula= equal 23 kW when I expect 25?" to the
participating components in three clicks.

Acceptance: tooltip appears within 200 ms; click navigates to
the formula tree; clicking a =#N= in the tree selects the
component on the canvas.

* Snapshots / save-state

** TODO G1. Snapshot save/load

Build on the existing overrides file. Add:

  Save snapshot → writes snapshots/<timestamp>.lisp
  Load snapshot → (reset-state) + (load …) the file

Snapshot captures: every registered component + its current
dynamic source (lambda installed via set-meter-power /
set-solar-sunlight serialises as the lisp expression).

=swctl snapshot save NAME= / =load NAME= for headless usage.

Useful for "I crashed my app at this state, reproduce it
later." Fits naturally with the existing
=load_overrides= + per-microgrid-id sidecar pattern.

* Stream stability

** TODO H1. Exponential WS reconnect backoff

Today /ws/events one-shots and the chrome flips to
"disconnected" forever. Port tradingsim's pattern:

  ws.onclose:  delay = min(prev * 2, 30s); reconnect after delay
  ws.onopen:   delay = 1s; setStatus("connected")

A developer running an overnight soak shouldn't lose the live
view because the laptop briefly slept.

Acceptance: kill -STOP / -CONT on the server makes the UI
reconnect within 30 s without manual refresh.
