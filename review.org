* High severity — real bugs

** H1. XSS gap in setpoint-event row rendering                       :TODO:
[[file:ui-assets/app.js::198][ui-assets/app.js:198]],
[[file:ui-assets/app.js::964][:964]]

#+begin_src js
const tag = ev.setpoint_kind.replace("_", " ");
const head = `<span class="sp-ts">${ts}</span> <span class="sp-tag">${tag}</span> <span class="sp-val">${ev.value}</span>`;
#+end_src

~ev.reason~ is escaped on the next line; ~tag~ and ~ev.value~ aren't.
Today the server only sends fixed strings and floats, so the gap
is theoretical — but defense-in-depth: every interpolation into
~innerHTML~ should go through ~escapeHtml~.

** H2. ~microgrids_create~ port allocation race                     :TODO:
[[file:src/ui/mod.rs::747][src/ui/mod.rs:747-748]]

~next_free_id~ and ~next_free_port~ run outside the registry lock.
Two concurrent create requests can pick the same id AND the same
port. The duplicate-key check on insert catches the id collision
(409), but two concurrent calls landing on different ids can still
both pick port 8810; the second tonic ~Server::serve~ then fails
silently inside its ~tokio::spawn~ and the loopback for that
microgrid sits "connecting" forever.

Fix: re-probe id + port inside the same lock that does the insert,
or wrap allocate-then-insert in a single critical section.

** H3. ~microgrids_create~ half-failed state ordering                :TODO:
[[file:src/ui/mod.rs::775][src/ui/mod.rs:775-784]]

Current order: insert into registry → call ~spawner~ →
~write_microgrid_stub~. If the stub write fails, we 500 but the
registry entry stays AND the spawner already started physics +
history sampler + gRPC server + loopback tasks. Next boot orphans
the live microgrid (no stub file to recreate it).

Fix: write the stub BEFORE the spawner. On stub-write failure,
roll back the registry insert and return without starting tasks.

** H4. ~Config::watch~ unwraps                                       :TODO:
[[file:src/lisp/mod.rs::870][src/lisp/mod.rs:870, 876]]

#+begin_src rust
let mut watcher = RecommendedWatcher::new(...).unwrap();
watcher.watch(Path::new(&self.filename), ...).unwrap();
#+end_src

inotify watch limit (~/proc/sys/fs/inotify/max_user_watches~
exhausted by a heavy IDE) or the config file vanishing mid-boot
panics this task. Binary keeps serving but hot-reload is dead
silently. Log + return on error.

** H5. ~overrides_path~ fallback to id 0                             :TODO:
[[file:src/lisp/mod.rs::682][src/lisp/mod.rs:682-689]]

#+begin_src rust
let mg_id = self.current_microgrid.read().unwrap_or_else(|| {
    self.microgrids.lock().keys().next().copied().unwrap_or_default()
});
#+end_src

When ~current_microgrid~ is None AND the registry is empty,
~unwrap_or_default()~ returns 0; the override path becomes
~microgrids/config.0.overrides.lisp~. The hard-error in ~Config::new~
makes the boot path unreachable, but a future
~reset-microgrid~-then-eval flow could trip it. Return a ~Result~
and refuse to write overrides without a resolved id.

** H6. ~set_main_meter~ failure mid-~make-meter~ leaves a partial meter :TODO:
[[file:src/lisp/make.rs::257][src/lisp/make.rs:257-260]]

#+begin_src rust
let h = register_with_modes(&w, meter, ...)?;
apply_initial_name(&w, id, a.name);
if a.main.unwrap_or(false) {
    w.set_main_meter(id).map_err(Error::invalid_argument)?;
}
#+end_src

A second ~:main t~ meter in the same microgrid hits the
~set_main_meter~ Err — but the meter is already in the registry,
named, and (if anyone followed up) connected to its successors.
Eval surfaces the error but the world is partially mutated.

Fix: probe the main-meter slot BEFORE ~register_with_modes~, or
roll back the registry insert on Err.

** H7. ~pre_tick~ hook holds the interpreter lock through the whole tick :TODO:
[[file:src/lisp/mod.rs::274][src/lisp/mod.rs:274-282]]

#+begin_src rust
let pre_tick_hook = Arc::new(move |w: &MicrogridSite| {
    let mut guard = hook_ctx.borrow_mut();   // global ctx lock
    for c in w.components() { c.refresh_inputs(&mut guard); }
    timer_handle.tick(&mut guard);
});
#+end_src

~hook_ctx~ is the ~SharedMut<TulispContext>~ — same lock
~/api/eval~ grabs. A long-running paste blocks every microgrid's
physics tick until eval finishes. ~auto_advance~ also grabs this
lock for its ~:on~ funcalls.

No quick fix — splitting ~refresh_inputs~ to an atomics-only path
that doesn't need the interpreter is a real refactor. Worth a doc
note in the hook installer until then.

** H8. ~microgrid_registered~ broadcast capacity 64 + Lagged handling :TODO:
[[file:src/lisp/mod.rs::174][src/lisp/mod.rs:174]],
[[file:src/ui/mod.rs::1830][src/ui/mod.rs:1830]]

A 64+ make-microgrid burst (mass-create script) lags the WS pump's
receiver. The pump's Lagged branch just ~continue~ — silently
dropping any registrations that fell off the back. Re-snapshot on
Lagged (re-read the registry, spawn forwarders for every entry
that lacks one) or bump capacity to 1024.

* Medium severity

** M1. ~start_timeout_loop~ cadence drift                           :TODO:
[[file:src/lisp/mod.rs::392][src/lisp/mod.rs:392]]

~tokio::time::sleep(Duration::from_millis(100))~ instead of
~interval(100ms)~ with ~MissedTickBehavior::Skip~. Under load the
actual delay per iteration drifts above 100 ms; timeout precision
follows. Mostly best-effort already; easy idiom fix.

** M2. Per-mg config files have no forward-reference safety          :TODO:
[[file:sim/common.lisp::68][sim/common.lisp]] +
[[file:src/lisp/mod.rs::1497][~load-microgrid-configs~ in src/lisp/mod.rs]]

~(load-microgrid-configs)~ loads ~microgrids/config.*.lisp~ in
sorted order. Nothing prevents ~config.2.lisp~ from referencing a
defvar or defun that only ~config.10.lisp~ introduces. Today each
per-mg file is expected to be self-contained, but the rule isn't
enforced. Either document the constraint prominently or sandbox
each file with its own lexical scope.

** M3. No graceful shutdown                                          :TODO:
Every tokio task (physics, history sampler, scenario auto-advance,
frequency driver, WS forwarders per session, gRPC servers, timeout
loop) runs until process exit. No ~CancellationToken~ plumbed. Ctrl-C
kills mid-tick — history rings lose the last second of samples, an
open CSV sink loses its trailing fsync. Production-grade fix; dev
tool is fine as-is. Flagging for visibility.

** A5. ~Config::new~ has delicate ordering invariants                :TODO:

Pre_tick installs AFTER eval_file (so the hook closure can capture
the timer handle), but the hook is installed retroactively on every
site that registered DURING eval. The ~make-microgrid~ defun reads
~pre_tick_slot~ to install on later sites. Frequency driver spawns
BEFORE eval. Timeout loop spawns BEFORE eval. Microgrid-registered
channel is created BEFORE eval but subscribed-to from WS later.

The constraints are real but invisible. Either add a comment block
at the top of ~Config::new~ listing them, or factor the bootstrap
into explicit phases (~prepare_state~, ~register_defuns~,
~spawn_long_lived_tasks~, ~eval_config~, ~install_post_eval_hooks~).

** A6. Bundle shared handles into a ~RuntimeHandles~ struct          :TODO:

~Config::new~ plumbs ~12 Arc-clones through the call. Define a
~RuntimeHandles~ struct (or similar) that carries ~ctx~, ~router~,
~current_microgrid~, ~id_allocator~, ~pre_tick_slot~,
~microgrid_registered~, ~grid_frequency~, etc. Pass that into each
~register_*~. Each fn picks the fields it needs. Side benefit:
fixes CI2's ~too_many_arguments~ on ~register_microgrids~ without
an ~#[allow]~.

** A7. "Snapshot under lock, fire callbacks outside" pattern         :TODO:

[[file:src/sim/scenarios.rs::334][~auto_advance_tick~]] and the start /
stop / jump siblings all replicate the same three-step dance: lock
registry, snapshot the transitions, drop lock, funcall lambdas
outside the lock. Worth a helper like ~transition_under_lock<T>(reg,
f) -> Vec<T>~ that callers thread their lambdas through.
